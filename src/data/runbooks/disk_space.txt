RUNBOOK: Disk Space Exhaustion

Symptoms:
- Datadog alert on system.disk.in_use exceeding 85% threshold
- Application errors: "No space left on device" in pod or node logs
- PostgreSQL write failures: "could not write to file" or "ERROR: could not extend file"
- Kubernetes events: "FailedMount" or "NodeHasDiskPressure" taint applied to a node
- Log shipping to S3 or Elasticsearch stopped (disk buffer full)
- Prometheus TSDB compaction failing or dropping samples

Common Causes:
1. Log files accumulating without rotation (missing logrotate config or fluentd backpressure)
2. PostgreSQL WAL accumulation due to a stuck replication slot
3. Container image layers and unused volumes piling up on nodes over time
4. Core dump files generated by crashing processes not being cleaned up
5. Temporary files from large data processing jobs not being deleted on completion
6. Prometheus or Elasticsearch data retention period set too long for available disk

Diagnostic Steps:
1. Identify which node is under disk pressure:
   kubectl get nodes -o wide
   kubectl describe node <node-name> | grep -A10 "Conditions:"

2. SSH to the node (or exec into a privileged pod) and find large directories:
   df -h
   du -sh /* 2>/dev/null | sort -rh | head -20
   du -sh /var/log/* | sort -rh | head -10

3. Check for stuck PostgreSQL replication slots consuming WAL:
   SELECT slot_name, active, pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)) AS lag
   FROM pg_replication_slots;

4. Look for large log files in the application pod:
   kubectl exec -n <namespace> <pod-name> -- du -sh /var/log/* /tmp/* 2>/dev/null | sort -rh | head -10

5. Check for dangling Docker images and volumes on the node:
   docker system df   # if docker is accessible on node
   crictl images | wc -l

6. Check Prometheus or Elasticsearch disk usage and retention settings

7. Look for core dump files:
   find /var/crash /tmp /core -name "core*" -size +100M 2>/dev/null

Remediation:
• Immediately free space by removing old log files (confirm they are not needed for debugging):
  find /var/log/app -name "*.log.*" -mtime +7 -delete

• Drop a stuck replication slot in PostgreSQL that is preventing WAL cleanup:
  SELECT pg_drop_replication_slot('<slot_name>');
  -- Only do this after confirming the consumer is no longer active

• Run Docker/containerd image prune on the node:
  kubectl debug node/<node-name> -it --image=busybox -- chroot /host docker system prune -f

• Delete completed or failed Kubernetes jobs that may have leftover ephemeral storage:
  kubectl delete jobs --field-selector status.successful=1 -n <namespace>

• Expand the PersistentVolume if on EBS/GCE PD:
  kubectl patch pvc <pvc-name> -n <namespace> -p '{"spec":{"resources":{"requests":{"storage":"100Gi"}}}}'

• Reduce Prometheus retention period as an emergency measure:
  Edit Prometheus config: --storage.tsdb.retention.time=7d (from 30d)
  kubectl rollout restart deployment/prometheus -n monitoring

Escalation:
- Page the infrastructure/platform on-call if disk usage exceeds 90% and the cause cannot be identified within 10 minutes
- Page the DBA immediately if PostgreSQL WAL accumulation is the cause and a replication slot cannot be safely dropped
- Escalate to the node pool team if NodeHasDiskPressure is causing widespread pod evictions
